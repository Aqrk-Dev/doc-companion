# doc-companion

A Claude Code plugin that keeps documentation attached to the code.

> 中文文档:[README.md](README.md)

## What is this

When AI helps you change code, documentation quietly goes stale — an interface changes, a field gets renamed, but the docs still describe the old version and nobody notices. doc-companion makes every change that touches a contract visible: touching a critical file requires you to declare its documentation impact first; each phase starts by reconciling any existing documentation drift down to zero; wrapping up automatically produces a change ledger, records the reconciliation outcome, and stamps it. It never writes documentation for you, and it never blocks you — every check is fail-open: on any problem it lets you through with a warning instead of getting in your way.

## Quick start

1. Run `/docc:init` once per repo — it interactively generates the mapping file and the hash baseline.
2. Day to day, you don't need to think about it — hooks handle reminders/interceptions automatically.
3. At the start of each phase run `/docc:preflight`; at wrap-up run `/docc:postflight`.

## What you'll run into day to day

- **Editing a mapped file** → a one-time anchor reminder; no action needed.
- **Editing a `critical` contract file** → intercepted once: declare this change's documentation impact in your message, replacing the `<!-- pending -->` placeholder in the declaration draft with your declaration, then retry as-is and it goes through — the same file in the same session won't be intercepted a second time.
- **Nearing the end of a session** → a one-line reminder that it's time to run `/docc:postflight`.
- **Stamping refused (`stampBlocked`)** → means there are still candidates that haven't been reconciled, or the mapping itself has a defect: finish reconciliation and re-stamp with `--verdict`, or fix the mapping; for a genuine exception case, `--force` explicitly skips this.

## Configuration at a glance

```json
{
  "version": 1,
  "config": {
    "ledgerDir": ".docc/LEDGER",
    "exclude": ["ent/", "**/*_pb.go"],
    "history": true,
    "historyLimit": 500
  }
}
```

- `version`: map.json's version number, generated by init — don't hand-edit it.
- `ledgerDir`: where the change ledger is stored, defaults to `.docc/LEDGER`.
- `exclude`: extra files/directories excluded from drift tracking.
- `history`: whether each stamp appends a line to `.docc/history.jsonl`, on by default.
- `historyLimit`: max lines kept in history.jsonl, defaults to 500, `0` means unlimited.

For exact semantics (pattern/anchor/exclude matching rules, etc.) see the [deep reference](docs/REFERENCE.md) (Chinese).

## CI / pre-commit (optional)

The `--check` flag can be used in a pre-commit hook or CI to check whether commits made outside a session introduced drift or mapping issues; exits 0 when clean, non-zero when there are problems.

```sh
node <plugin-path>/scripts/doc-preflight.mjs --cwd . --check
```

## FAQ and limitations

**Does it support Windows?** Yes — native Windows (including PowerShell without Git Bash) has been supported since 0.6.0.

**How do I use it in a monorepo subdirectory?** Start your session inside the subpackage that contains `.docc/` (or one of its subdirectories).

**Is it safe to run multiple sessions in parallel?** Each session writes its own declaration draft; reconcile them manually at postflight. Duplicate interceptions under concurrency are rare, and a retry gets you through.

**What should I watch for when upgrading?** See [CHANGELOG.md](CHANGELOG.md); if stamping is refused after an upgrade, fix the mapping defects first — `--force` for genuine exception cases.

**Why doesn't `pattern` support `!`/`{a,b}` exclusion syntax?** That's an intentionally minimal design; use `config.exclude` for exclusion needs.

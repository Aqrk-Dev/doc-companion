## 2. 详细设计

### 2.1 `.docc/` 独立数据目录

**布局**:

```
.docc/
├── map.json        # 原 docs/DOC-MAP.json(契约登记,人维护)
├── hashes.json     # 原 docs/DOC-MAP.hashes.json(共识基线,机器写)
├── history.jsonl   # 原 docs/DOC-MAP.history.jsonl(观测数据,机器写)
└── LEDGER/         # 台账默认位置(config.ledgerDir 仍可配,想留 docs/LEDGER 的项目自行配置)
```

**常量**(`hooks/_shared.mjs`):`DOC_MAP_FILE=".docc/map.json"`、`HASHES_FILE=".docc/hashes.json"`、`HISTORY_FILE=".docc/history.jsonl"`;`DEFAULT_EXCLUDES` 增加 `".docc/"`。`loadDocMap` 只读新位置,无旧位置回退。

**整类简化**(随目录排除自然获得,删除既有特判):
- preflight 源侧枚举中 `f !== HASHES_FILE && f !== HISTORY_FILE` 特判删除;
- `docs[].file` 自引用逐文件特判(selfref)简化为:`relDoc` 以 `.docc/` 开头 → mapIssue `docc 数据目录内文件不可登记为文档锚点: <relDoc>`;
- 移除 0.2.1 旧基线非规范键兼容垫(`recDocs["./"+df] / ["/"+df]`)——重新 init 后基线键必然规范;
- `docs[].file` 前导 `/` 按仓库根相对处理的语义**保留**(这是约定,不是兼容垫)。

**内置自守护(新设计点)**:旧方案靠把 DOC-MAP 登记为 critical 条目自守护,新布局下 `.docc/` 被恒排除、登记失效。改为契约门 `doc-map-gate.mjs` **内置**守护:编辑目标 `rel` 为 `DOC_MAP_FILE`/`HASHES_FILE`/`HISTORY_FILE`(即 `.docc/` 下三个数据文件的完整仓库相对路径)之一时,恒视为 critical 拦截(锚点显示为 `(.docc 数据文件)`),无需登记、不可关闭——该判断必须在 `matchEntries` **之前**短路(否则 `.docc/` 目录排除会先返回空匹配);`.docc/LEDGER/` 不拦(写台账是 postflight 正常动作)。remind/stop-remind 对 `.docc/` 维持排除(不提醒)。init SKILL 删除"自守护条目"指引。

**有意接受**:hashes/history 被会话外手工篡改无实时防护(原"登记 pattern 触发 gate"路径随布局取消)——威胁模型内(能改仓库者也能重盖章),靠 git 历史审计,写入已知限制。

### 2.2 exclude 模式级排除

`config.exclude` 成员含 `*` 或 `?` → 按极简 glob 编译(复用 `globToRegExp`)整路径匹配;否则维持目录前缀语义(归一化同 0.2.0)。两类可混写:

```json
{ "config": { "exclude": ["ent/", "**/*_pb.go", "**/*.gen.go"] } }
```

**实现**:`getExcludes(docMap)` 返回 `{ prefixes: string[], regexps: RegExp[] }`(内部 API,破坏性无碍);`isExcluded(rel, ex)` 两类都查;`matchEntries`/preflight 调用点随之调整。glob 编译失败的成员静默忽略(fail-open)。

**顺手加固**:`globToRegExp` 入口折叠连续 `**/`(`pattern.replace(/(?:\*\*\/)+/g, "**/")`)——消除审计候选中"连续 `**/` 嵌套量词灾难性回溯"的暴露面,pattern 与 exclude 同时受益。

### 2.3 盖章一致性(范围改源 + 盖章门)

**范围改源**(postflight SKILL 重写第 1-2 步):核对清单不再来自 `git diff <diffBase>...HEAD`,直接以机检报告的 `driftCandidates` + `docDrift` 为准(口径统一为"相对基线的差异",与盖章范围天然一致)。`config.diffBase` 字段删除。

**盖章门**(`doc-preflight.mjs`):`--stamp` 时统计**真漂移** = "有基线记录且哈希失配" + "基线有记录但文件已消失"(**首次纳管**——无基线记录——不算),加上 `docDrift` 全部。真漂移数 > 0 且未传 `--force` → 拒绝盖章:`stamped:false`,mapIssue `存在未处置漂移(真漂移 N 源文件 + M 文档),先完成语义对拍;例外登记场景用 --force`。`--force` 跳过该门。init 首次建基线全为首次纳管,天然通过。拒绝盖章仍 exit 0(fail-open 红线:这是防错门,不阻塞任何开发动作)。

### 2.4 history 携带语义判定(--verdict)

**CLI**:`--verdict '<json>'`,仅与 `--stamp` 联用。值为对象,只接受四个已知键的非负数字:`formatting`(格式化误报)/`docLag`(文档滞后)/`codeViolation`(代码违约)/`other`。解析失败或缺失 → `verdict: null` + warnings `盖章未附对拍判定(--verdict),历史观测缺口`。

**history 行 schema(0.3.0 起)**:

```json
{"ts":"<ISO8601>","driftCandidates":0,"docDrift":0,"mapIssues":0,"warnings":0,"stamped":true,"forced":false,"verdict":{"formatting":1,"docLag":2,"codeViolation":0,"other":0}}
```

(`forced` 记录是否 `--force` 盖章。)自此可直接统计"格式化误报累计出现次数"——P0 结构指纹的触发条件可观测。

**SKILL 接线**:preflight 第 2 步语义对拍要求按四类记数;postflight 盖章命令模板带 `--verdict`(值来自对拍结论);init 建基线用 `--verdict '{}'`。

## 4. 红线复核

- **fail-open**:盖章门拒绝仍 exit 0 且只影响盖章动作本身;`--verdict` 缺失仅软告警;glob 编译失败静默忽略;内置守护走既有 gate 流程(状态不可写降级放行不变)。
- **同会话同文件只拦一次**:内置守护复用 `state.gated`,语义不变。
- **零 npm 依赖 / 零项目耦合**:不变。
- **契约冻结靠人为裁决**:盖章门强化(而非替代)人为裁决——它只拦"没走完流程的盖章",`--force` 保留人的最终决定权。
